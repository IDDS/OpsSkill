# AI 辅助论文写作范式：提示词模板

> 本文件包含论文写作流水线各阶段的即用提示词。
> 使用时复制粘贴到你的 AI 助手（Copilot / ChatGPT / Claude 等），填入 `[方括号]` 中的内容即可。
> 从 OpsSkill 论文完成过程中提炼。

---

## 阶段 1：文献调研与领域洞察

### 提示词 1.1 —— 系统性文献搜索
```
我的研究方向是 [1-2 句话描述研究方向]。

请帮我：
1. 生成 5 组搜索关键词（含同义词和英文变体），用于 Google Scholar / DBLP / arXiv 搜索
2. 建议应该重点关注的顶级会议和期刊（根据 [CCF-A/B 或领域顶会]）
3. 推荐用 Connected Papers / Litmaps 扎入的 3-5 篇种子论文（如果你知道这个领域）
```

### 提示词 1.2 —— 文献精读与优缺点提取
```
请阅读以下论文（或摘要），并提取结构化信息：

论文：[粘贴论文标题/摘要/全文]

请填写：
1. 核心方法（一句话）
2. 评估方式（数据集/指标/基线）
3. 主要优点（2-3 条）
4. 明确局限（作者自述 + 你的判断，各 1-2 条）
5. 代码/数据是否可复现
6. 与我的研究方向的关系（直接竞品 / 方法借鉴 / 问题相关 / 设计启发）
```

### 提示词 1.3 —— 领域缺口分析
```
我已经调研了以下论文：
[P01] [XX, 2024] — 做了...，局限是...
[P02] [XX, 2023] — 做了...，局限是...
[P03] ...
...

请帮我：
1. 归纳这些工作的共同局限，分类为 2-3 个高频缺口（gap）
2. 对每个缺口评估：实用性、可行性、可度量性
3. 按优先级排序，建议最值得攻先的方向
4. 画出简化版的领域知识地图（哪些子方向已饱和、哪些有空白）
```

## 阶段 2：创新点提炼与研究定位

### 提示词 2.1 —— 从缺口到创新点
```
我发现该领域的主要缺口是：
1. [GAP_1：具体描述]
2. [GAP_2：具体描述]
3. [GAP_3：具体描述]

我初步的想法是：[1-2 句话描述你的思路]

请帮我：
1. 将想法升级为明确的创新点陈述（格式：「A 做不到 X，因为缺少 Y，我们通过 Z 来解决」）
2. 拆解为 3-5 条可度量的贡献
3. 对每条贡献，建议验证手段（实验 / 证明 / 案例）
4. 评估创新点的强度：是增量改进还是范式转变？
```

### 提示词 2.2 —— 研究定位与目标期刊匹配
```
我的创新点是：[1-2 句话]
研究类型是：[系统型 / 理论型 / 应用型 / 实证型]
涉及的领域：[SE / AI / Systems / Security / ...]

请建议：
1. 3-5 个适合的目标期刊/会议，说明各自偏好
2. 每个期刊/会议的审稿重点（新颖性 vs. 实验 vs. 理论 vs. 实用性）
3. 我的工作最匹配哪个，为什么
4. 针对所选期刊/会议，论文应该强调哪些方面
```

### 提示词 2.3 —— 方法草案验证
```
我计划用以下方法解决 [GAP]：
[2-3 句描述方法草案]

请扮演「魔鬼辩护人」，找出这个方案的潜在问题：
1. 技术可行性问题（能不能做到？）
2. 新颖性问题（和已有工作的区别够大吗？）
3. 评估问题（怎么证明它有效？）
4. 范围问题（太大做不完？太小不值得发？）
并建议如何应对每个问题。
```

## 阶段 3：问题定框与项目启动

### 提示词 3.1 —— 起草问题陈述
```
我的研究要解决的真实痛点是：
[用 2-3 句话描述真实问题]

现有方法不行的原因：
1. [局限 1]
2. [局限 2]
3. [局限 3]

请帮我：
1. 写一个形式化的一句话问题陈述
2. 定义 3-5 条可度量的贡献
3. 提出 3-5 个研究问题（RQ），每个直接映射到实验
4. 明确范围内和范围外的内容
```

### 提示词 3.2 —— 贡献-实验映射
```
我的贡献是：
C1: [贡献 1]
C2: [贡献 2]
C3: [贡献 3]
...

对每条贡献，请建议：
- 用哪种实验类型证明（对比 / 消融 / 案例分析 / 形式化证明）
- 用什么指标衡量
- 负面结果会是什么样子
```

### 提示词 3.3 —— 初始化项目结构
```
我要开始一个叫 [项目名] 的研究项目。
主题是：[一句话描述]。

请帮我创建项目目录结构，包括：
- docs/（论文草稿、参考文献）
- src/ 或 [包名]/（核心实现）
- experiments/（任务配置、故障注入清单）
- scripts/（实验运行器、图表生成器）
- results/（原始数据、figures/、tables/）
- configs/（环境配置）
- skills/ 或 models/（学习到的产物）

同时创建 README.md，包含项目概述、环境搭建、复现步骤。
```

### 提示词 3.4 —— 搭建实验基础设施
```
我有一个 [平台] 环境（[描述，如：3 节点 Kubernetes 集群 / AWS ECS / OpenStack 等]）。
目标应用是 [应用描述]。
我想注入 [故障类型] 来测试我的方法。

请帮我：
1. 编写目标应用的部署清单
2. 编写故障注入配置（如 Chaos Mesh YAML / Litmus 配置）
3. 编写一键部署脚本
4. 编写一键清理脚本
```

## 阶段 4：方法形式化

### 提示词 4.1 —— 形式化定义
```
我的方法 [方法名] 的工作方式（非正式描述）：
[用 5-10 句话描述你的方法]

请帮我形式化为：
1. 环境/输入定义（用数学元组表示）
2. 核心数据结构（方法产出什么？）
3. 目标函数（在优化什么？）
4. 约束条件（什么必须始终成立？）
5. 流水线图描述（输入 → 步骤1 → 步骤2 → ... → 输出）
```

### 提示词 4.2 —— 命题与证明
```
我声称我的方法有这个性质：
[非正式声称，如"门控机制能降低不安全操作率"]

机制原理是：[描述机制]

请帮我：
1. 写成带清晰假设的形式化命题
2. 写严谨的证明
3. 加一个"注"段落，说明需要什么假设、什么时候假设可能不成立
```

### 提示词 4.3 —— 架构图描述
```
我的系统有这些组件：
1. [组件 1]：输入 [X]，输出 [Y]
2. [组件 2]：输入 [X]，输出 [Y]
...

数据流是：[组件 1] → [组件 2] → ...
当 [条件] 时，从 [组件 X] 反馈回 [组件 Y]。

请用 matplotlib/tikz/drawio 生成框架总览图，
标清标签、箭头，用虚线框区分 [阶段 A] 和 [阶段 B]。
```

## 阶段 5：代码实现

### 提示词 5.1 —— 从架构图到代码骨架
```
我的方法架构如下：
[描述模块及其输入输出，或贴架构图]

使用语言/框架：[Python / Go / ...]
目标平台：[Kubernetes / AWS / 本地 / ...]

请帮我：
1. 设计项目目录结构（模块划分与架构图一一对应）
2. 定义核心接口/类/数据结构（变量名与论文公式符号对应）
3. 建议实现顺序（核心模块优先）
4. 写一个冒烟测试脚本，验证端到端流程可跑通
```

### 提示词 5.2 —— 核心模块实现
```
我需要实现 [MODULE_NAME] 模块。

功能：[WHAT_IT_DOES]
输入：[INPUT_FORMAT]
输出：[OUTPUT_FORMAT]
对应论文中的：[公式编号或章节]
依赖：[其他模块或外部服务]

请实现这个模块，要求：
1. 代码中的变量名与论文符号对应
2. 关键步骤有注释说明
3. 包含单元测试
4. 配置参数外置
```

## 阶段 6：实验设计

### 提示词 6.1 —— 基线选择
```
我的方法是用于 [任务领域] 的 [方法名]。
核心创新点：[列出创新点]。

请建议 4-5 个基线，满足：
1. 代表最简单的方法（下界）
2. 代表当前最强的推理方法
3. 代表当前最强的自改进方法
4. 代表知识检索方法
5.（可选）代表工业工具

对每个基线，说明为什么它是公平的对比，以及它测试什么具体假设。
```

### 提示词 6.2 —— 消融设计
```
我的方法有这些核心组件：
1. [组件 A]：做 [功能 A]
2. [组件 B]：做 [功能 B]
3. [组件 C]：做 [功能 C]

请设计消融变体：
- 分别独立移除每个组件
- 对每个变体，预测哪个指标会下降、为什么
- 建议如何命名（如 "A1-no-XX"）
```

### 提示词 6.3 —— 统一指标设计
```
对比中不同方法有不同能力：
- 方法 A 能做：[能力 A]
- 方法 B 能做：[能力 B]
- 我的方法能做：[全部能力]

当前分别打分可能不公平。
请设计统一评估指标，满足：
1. 对所有方法用相同公式
2. 覆盖所有能力维度
3. 方法不具备的能力得 0 分（不是惩罚，是能力缺失）
4. 包含权重敏感性分析计划（至少 5 种权重配置）
```

### 提示词 6.4 —— 对抗/优化实验
```
我的方法包含优化循环，基于 [反馈] 改进 [产物]。
优化机制将 [失败类型] 映射到 [结构化编辑]。

请设计 5-6 个对抗场景：
1. 每个触发一种特定失败类型
2. 每个测试一条不同的优化规则
3. 包含优化前后对比
4. 同时衡量分数提升和结构变化（如前置条件数量、验证信号数量）
```

## 阶段 7：实验执行与数据流水线

### 提示词 7.1 —— 实验运行脚本
```
我有 [N] 种方法 × [M] 个任务 = [N×M] 组实验。

每组实验需要：
1. [准备步骤，如注入故障]
2. [运行步骤，如执行方法]
3. [评估步骤，如运行验证器]
4. [清理步骤，如移除故障]

请写一个 Python 实验运行器：
- 遍历所有 方法×任务 组合
- 每组保存结构化 JSON 结果
- 有容错能力（可从中断处恢复）
- 带时间戳的进度日志
```

### 提示词 7.2 —— 自动生成图表
```
我的实验结果在 [路径] 下，格式是 [如 JSON]。
包含的字段：[描述字段]。

请写 Python 脚本生成：
1. 柱状图：所有方法对比（主结果）
2. 分组柱状图：消融实验
3. 雷达图：跨域对比
4. 热力图：方法 × 任务得分
5. 散点图：效率（执行时间）
6. 前后对比图：优化效果
7. 瀑布图：改进分解

每张图保存 PDF（给 LaTeX 用）和 PNG（预览用）。
同时生成 LaTeX 表格文件（.tex），可直接 \input{} 导入。
```

## 阶段 8：论文撰写

### 提示词 8.1 —— 逐节起草
```
我正在写论文的 [章节名] 部分。

背景：
- 论文标题：[标题]
- 本节相关的核心贡献：[贡献]
- 主要结果/证据：[结果]

已有草稿（如有）：
[粘贴已有文字，或留空]

请按以下规则写这一节：
1. 第一句话就是本节的核心信息
2. 每段有一个主题句 + 支撑证据
3. 每个声称引用图/表/公式
4. 末尾过渡到下一节
5. 目标长度：[N] 字
6. 风格：[期刊/会议，如 IEEE conference、ACM journal]
```

### 提示词 8.2 —— 引言公式
```
我的论文：
- 标题：[标题]
- 问题：[问题]
- 现有方法为何失败：[缺陷 1]、[缺陷 2]、[缺陷 3]
- 我们的方案：[方法名]，核心做法是 [核心思路]
- 核心数据：[主要数字]
- 贡献：C1, C2, C3, ...

请按以下结构写引言：
1. 第 1 段：背景与重要性（引用 [综述论文]）
2. 第 2 段：现有方法做了什么、有什么局限
3. 第 3 段：我们的方法与核心洞察
4. 第 4 段：编号列表形式的贡献
```

### 提示词 8.3 —— 相关工作组织
```
我的论文主题是 [主题]。最相关的先前工作分为以下类别：
1. [类别 1]：论文 [列表]
2. [类别 2]：论文 [列表]
3. [类别 3]：论文 [列表]
4. [类别 4]：论文 [列表]

对每个类别，请写一段：
1. 总结这些工作做了什么（2-3 句）
2. 找出它们与我们工作相关的共同局限（1 句）
3. 说明我们的方法如何弥补这一局限（1 句）

最后用对比表或总结段落定位我们的方法。
```

### 提示词 8.4 —— 讨论部分
```
我的实验结果表明：
- [关键发现 1]
- [关键发现 2]
- [关键发现 3]

请写讨论部分，包含这些子节：
1. 为什么 [组件 A] 有效（超越数字的深层分析）
2. 权衡分析（如安全性 vs. 效率）
3. [组件 B] 的价值（优化/学习）
4. 评估公平性讨论
5. 局限性（至少 4 条具体、诚实的局限）
6. 效度威胁（内部、外部、构造）
```

### 提示词 8.5 —— 案例分析写作
```
我想写一个详细的案例分析，展示方法的端到端流程。

场景：[描述]
发生的步骤：
1. [步骤 1 及输出]
2. [步骤 2 及输出]
...
对比：基线方法做了 [基线做了什么]，失败原因是 [原因]。

请写成案例分析子节，包含：
1. 场景描述（粗体标题）
2. 逐步流程展示，带代码/输出块
3. 高亮关键设计洞察
4. 在相同场景下与基线的显式对比
```

## 阶段 9：证据-声称对齐

### 提示词 9.1 —— 声称-证据审计
```
以下是我论文中的所有声称：
[粘贴声称列表 / 关键句子]

以下是所有证据产物：
- 表：[列表]
- 图：[列表]
- 证明：[列表]

请创建映射表：
| 声称 | 证据 | 强度（强/中/弱） | 有缺口？ |

标出任何没有强证据支撑的声称。
```

## 阶段 10：自审稿与模拟评审

### 提示词 10.1 —— 模拟审稿人
```
请扮演 [期刊/会议，如 ICSE、NeurIPS、SIGCOMM] 的严格审稿人。

阅读以下论文（或摘要+方法+实验摘要）：
[粘贴内容]

请给出：
1. 三个优点
2. 三个缺点（严格但公正）
3. 两个你会问作者的问题
4. 一个"致命缺陷"（如有），或"无"
5. 一个能加强论文的额外实验建议
```

### 提示词 10.2 —— 版本一致性检查
```
我有两个版本的论文：
- 版本 A：[语言 A / 格式 A]，[N_A] 行
- 版本 B：[语言 B / 格式 B]，[N_B] 行

请做逐节对比，报告：
1. A 中有但 B 中缺失的章节
2. A 中有但 B 中缺失的图/表
3. A 中完整但 B 中被缩写的证明
4. 任何被大幅缩短的内容
5. 内容保留百分比
```

### 提示词 10.3 —— Rebuttal 草案
```
以下是模拟审稿人提出的意见：
[粘贴模拟审稿意见]

请帮我写 rebuttal 回复，要求：
1. 对每个缺点/问题透一回复
2. 格式：「感谢建议。我们的回应是…」
3. 对于能补实验的，说明补什么实验、预期结果
4. 对于不能补的，说明为什么不影响核心结论
5. 对于写作问题，给出修改后的具体文本
```

### 提示词 10.4 —— 修改循环定位
```
以下是审稿意见和我的 rebuttal 草案：
[粘贴]

请帮我：
1. 将每条修改任务映射到范式的具体阶段（阶段 2–12）
2. 按优先级排序（致命缺陷 > 实验不足 > 形式化 > 写作）
3. 估算每条修改的工作量（小 / 中 / 大）
4. 生成修改任务清单（可作为 TODO list）
```

## 阶段 11：排版与提交

### 提示词 11.1 —— 期刊/会议格式转换
```
我有一篇 [源格式，如 ctexart、普通 LaTeX] 的论文。
需要转换为 [目标格式，如 IEEE 双栏、ACM sigconf]。

要求：
- 使用 [模板类，如 IEEEtran.cls]
- 所有图表适配栏宽/页宽
- 公式不溢出
- 参考文献样式：[样式，如 IEEEtran.bst]
- 已知字体问题：[列出，如果有]

请创建完整的转换后 .tex 文件，保留全部内容。
```

### 提示词 11.2 —— 编译修复循环
```
我的 LaTeX 编译产生以下错误/警告：
[粘贴日志片段]

相关的 .tex 文件段落：
[粘贴相关行]

请诊断每个问题并提供修复方案。
优先级：错误 > 溢出 > 警告。
```

### 提示词 11.3 —— 仓库准备与发布
```
我想把项目发布到 GitHub 公开仓库。

项目路径：[路径]
包含内容：[描述]

请帮我：
1. 创建适合 [语言/框架] 的 .gitignore
2. 检查 README.md 是否完整
3. 检查是否有敏感数据（API 密钥、密码、内网 IP）需要移除
4. 初始化 git、提交、创建 GitHub 仓库、推送
```

## 元提示词：论文冲刺
```
我想在 [N] 天内完成一篇完整的研究论文。
```
