# OpsSkill: 面向远程 Kubernetes 环境的自治运维技能学习框架

---

## 摘要

随着大语言模型在工具调用与自治 Agent 中表现出越来越强的能力，利用其辅助 DevOps 与 SRE 场景的自动化运维已成为一个重要研究方向。然而，现有方法大多将运维任务建模为自由形式的命令生成或交互式工具调用过程，缺乏对运维任务中前置条件、动作约束、成功判据、回滚机制与风险边界的统一建模，因而难以在真实基础设施环境中实现安全、可靠且可复用的自动化执行。

本文提出 OpsSkill，一个面向远程 Kubernetes 环境的自治运维技能学习框架。OpsSkill 将任务卡自动编译为结构化的技能表示，包含前置条件、动作序列、隐藏验证器、回滚计划与风险标签；在执行阶段，通过策略约束与能力探测机制控制技能的实际落地；在优化阶段，基于结构化执行报告与失败签名对技能进行持续改进。与仅依赖命令返回结果的评估不同，OpsSkill 进一步设计了面向真实远程集群的多信号隐藏验证机制，综合对象状态、事件、日志与服务健康信息，对技能是否真正完成运维目标进行判定。

我们在真实远程 Kubernetes 集群（3 节点，K8s v1.33.6）上，通过 Chaos Mesh 注入 CPU 压力、内存压力和网络延迟三类故障，对 8 个运维任务进行了系统评估。实验对比了直接命令生成、ReAct 风格 Agent、Reflexion 风格优化方法及模板检索四种基线，并通过消融研究量化了各组件的贡献。结果表明，OpsSkill 在综合评分上达到 90%，分别超过直接命令生成（25%）、ReAct（40%）、Reflexion（40%）和模板检索（25%）达 50-65 个百分点，同时保持零不安全动作率。消融研究表明类型化技能 IR 贡献了 65% 的评分提升，隐藏验证贡献了 30%，策略门控以 1% 的评分代价将不安全动作率从 12% 降至 0%。

**关键词**：自治运维、Kubernetes、大语言模型、技能学习、安全执行

## 1 引言

随着云原生系统和微服务架构在工业界的广泛部署，DevOps 与 SRE 团队需要持续处理大量重复但高风险的运维任务，例如故障诊断、配置修复、服务恢复、扩缩容调整以及权限或依赖异常处理。这些任务通常依赖运维人员对系统状态、工具链、故障模式以及组织约束的综合理解，具有明显的状态依赖性、环境异构性和执行风险。近年来，大语言模型在推理、代码生成与工具调用方面的快速进展，使得构建能够辅助甚至部分自动完成运维任务的自治 Agent 成为一个极具吸引力的研究方向。

然而，现有方法距离真实可用的自治运维仍存在显著差距。一方面，许多方法将运维任务简化为自由形式的命令生成或交互式工具调用问题，忽视了运维动作本身具有强约束和高代价的特点。对于真实生产或准生产环境而言，错误的命令序列不仅会导致任务失败，还可能带来跨命名空间影响、服务中断甚至不可逆的数据损坏。另一方面，现有评测往往依赖可见的命令返回值或单一检查项来判断任务是否完成，但在真实系统中，"命令执行成功"并不等价于"服务真正恢复"。例如，Pod 被成功重启并不意味着依赖链路恢复，Deployment rollout 完成也不意味着用户侧错误率已经下降。因此，自治运维系统不仅需要会"调用工具"，更需要具备安全执行、隐藏验证与失败后优化的能力。

为解决上述问题，本文提出 OpsSkill，一个面向远程 Kubernetes 环境的自治运维技能学习框架。OpsSkill 的核心思想是：将运维任务从一次性的命令生成问题提升为结构化技能学习问题。具体而言，我们将一个运维任务编译为带类型的技能表示，包含前置条件、动作序列、成功判据、回滚计划和风险标签；在执行层面，设计策略约束与能力探测机制，对技能进行门控执行；在评测层面，引入多信号隐藏验证器，综合对象状态、事件、日志与服务健康信息判断任务是否真正完成；在优化层面，利用结构化执行报告和失败签名持续改进技能质量。

本文做出如下贡献：

1. 提出面向运维任务的 **Typed Ops Skill IR**，将任务卡编译为包含前置条件、动作序列、成功判据、回滚计划与风险标签的结构化技能表示。
2. 设计面向远程 Kubernetes 的**策略门控执行框架**，在技能执行前进行能力探测、范围约束与风险检查。
3. 提出面向真实远程环境的**隐藏多信号验证机制**，将可见检查与隐藏检查相结合。
4. 提出基于结构化执行报告与**失败签名的技能优化机制**，使系统能够从失败中持续学习。
5. 在真实远程 Kubernetes 集群上进行了系统的基线对比和消融实验，验证了所提方法的有效性。

## 2 相关工作

现有与 OpsSkill 最相关的研究主要分为四类：AIOps 与事故知识挖掘、微服务诊断与根因定位、运行手册/故障排查指南自动化，以及通用 Agent 工具调用评测与安全治理。

**AIOps 与事故知识挖掘。** 第一类工作关注如何从真实云服务事故与运维调查中提取结构化知识。DeCaf 将大规模云服务中的性能问题诊断与分诊建模为一个连续流程，后续工作进一步从 incident 调查记录中挖掘根因知识或从事故文本中抽取结构化运维知识。这些方法大多停留在知识抽取层面，尚未将知识编译为带有前置条件、回滚与成功判据的可执行技能。

**微服务诊断与根因定位。** 第二类工作聚焦细粒度性能诊断、可解释故障定位以及基于异构图或不完整指标的根因分析。这些研究主要解决"找出问题在哪里"，而不是"如何在真实受限环境中安全地执行修复动作并验证修复是否有效"。

**运行手册自动化。** 第三类工作与 OpsSkill 最接近，包括 AutoTSG、incident copilot 和 agentic support automation。这些研究说明将运维经验转化为可操作计划是可行的，但通常缺乏统一的结构化 skill 表示，也很少在真实远程 Kubernetes 环境中系统考察安全门控和隐藏验证。

**Agent 工具调用与安全治理。** 第四类工作涉及 Agent 在工具调用、多步交互中的能力评测与安全治理机制。这些工作为 OpsSkill 提供了评测和安全设计上的背景，但要么聚焦通用工具调用，要么停留在原则性讨论。

## 3 问题定义

### 3.1 环境与任务卡

我们用 $E=(\mathcal{S}, \mathcal{O}, \mathcal{U})$ 表示远程运维环境，其中 $\mathcal{S}$ 为状态空间，$\mathcal{O}$ 为可观测信号空间，$\mathcal{U}$ 为工具动作空间。任务卡定义为 $x = (d, c, g, \Omega, \mathcal{T})$，其中 $d$ 为任务描述，$c$ 为上下文，$g$ 为目标状态，$\Omega$ 为约束条件，$\mathcal{T}$ 为可用工具集合。

### 3.2 环境能力画像

$$\kappa(E) = (\kappa_{rbac}, \kappa_{resource}, \kappa_{crd}, \kappa_{network}, \kappa_{obs})$$

### 3.3 技能表示

$$S = (P, A, V, R, \rho, \eta)$$

其中 $P$ 为前置条件，$A$ 为动作序列，$V=(V_{vis}, V_{hid})$ 为验证器，$R$ 为回滚计划，$\rho$ 为风险标签，$\eta$ 为实例化参数。技能合法性约束：

$$\Gamma(S) = \mathbb{I}[|A| > 0] \cdot \mathbb{I}[|V_{vis}| > 0] \cdot \mathbb{I}[\text{well-typed}(P,A,R,\eta)] = 1$$

### 3.4 技能编译-执行-优化闭环

$$x \xrightarrow{f_\theta} S \xrightarrow{g_\psi} \tau \xrightarrow{h_\phi} y \xrightarrow{u_\omega} S'$$

### 3.5 优化目标

$$\max_{\theta,\psi,\omega} \; \mathbb{E}[Y_{hid}] - \lambda_1 \mathbb{E}[U] - \lambda_2 \mathbb{E}[B] - \lambda_3 \mathbb{E}[R_b] - \lambda_4 \mathbb{E}[C] - \lambda_5 \mathbb{E}[T]$$

subject to $\Pr(B > b_{max}) \le \delta$.

## 4 方法

### 4.1 方法概览

OpsSkill 的核心实现原则：**skill 生成、验证、优化和调度四个环节均允许 LLM 或 agent 参与，但所有高风险执行都必须经过受控执行器。** 五个核心创新：

1. **约束感知技能编译**：将任务卡与环境能力画像联合映射为带类型的技能对象
2. **分层技能调度智能体**：由 manager agent 在三阶段之间进行 skill 选择
3. **风险预算门控执行**：将运维动作建模为受 blast-radius 约束的策略执行问题
4. **隐藏多信号验证**：区分技能可见验证与隐藏验证
5. **结构化失败驱动优化**：从结构化执行报告中提取失败签名进行技能更新

### 4.2 约束感知技能编译

给定任务卡 $x$ 与环境能力画像 $\kappa(E)$，技能编译器求解：

$$f_\theta(x, \kappa(E)) = \arg\max_{S \in \mathcal{S}_k} \, \mathcal{L}_{task}(S; x) - \lambda_r \mathcal{R}_{risk}(S, \kappa(E)) - \lambda_s \mathcal{R}_{schema}(S)$$

### 4.3 风险预算门控执行

#### 4.3.1 风险函数

$$\mathcal{B}(S, E) = w_1 b_{scope} + w_2 b_{mutation} + w_3 b_{privilege} + w_4 b_{rollback}$$

#### 4.3.2 门控器

$$g_\psi(S, \kappa(E), \Omega) = \mathbb{I}[P(S, E)=1] \cdot \mathbb{I}[\mathcal{B}(S,E) \le b_{max}] \cdot \mathbb{I}[\text{policy}(S,\Omega)=1]$$

**命题 1**（门控安全性）：若门控器只允许 $\mathcal{B}(S,E) \le b_{max}$ 的技能执行，则 $\hat{U}_{gated} \le \hat{U}_{ungated}$。

### 4.4 隐藏多信号验证器

$$V_{hid}(\tau) = \sum_{j=1}^{M} \alpha_j v_j(\tau)$$

**假阳性差距**：$\Delta_{fp} = \mathbb{E}[V_{vis}(\tau)] - \mathbb{E}[V_{hid}(\tau)]$

### 4.5 结构化失败驱动优化

$$u_\omega(S, r(\tau)) = \Pi_{\mathcal{S}_k}\big(S + \Delta_{\sigma}(S)\big)$$

**命题 3**（优化收敛性）：若失败签名映射 $\Delta_\sigma$ 能在期望上减少相应失败模式再发概率，则 $\mathbb{E}[\mathcal{L}_{fail}(S_{t+1})] \le \mathbb{E}[\mathcal{L}_{fail}(S_t)]$。

### 4.6 完整算法

1. 输入任务卡 $x$、环境 $E$ 与约束 $\Omega$
2. 探测环境能力 $\kappa(E)$
3. 编译候选技能 $S = f_\theta(x, \kappa(E))$
4. 检查合法性 $\Gamma(S)$
5. 策略门控 $g_\psi(S, \kappa(E), \Omega)$
6. 若阻断则输出报告；否则执行得到轨迹 $\tau$
7. 验证：$V_{vis}(\tau)$ 与 $V_{hid}(\tau)$
8. 生成报告 $r(\tau)$ 与失败签名 $\sigma$
9. 优化：$S' = u_\omega(S, r(\tau))$
10. 写回 skill bank

## 5 实验

### 5.1 实验环境

**集群配置。** 实验在 3 节点 Kubernetes 集群上进行（K8s v1.33.6，Docker v29.1.3），节点 `lsy-1`/`lsy-2`/`lsy-3` 各配备 8 核 CPU 与 32 GB 内存。通过 SSH 跳板远程访问：本地 → 跳板 (222.200.180.102) → 集群主节点 (10.10.3.110)。

**目标应用。** 基于 `nginx:1.27-alpine` 的单副本 Deployment（`demo-app`），配置资源配额（100m~500m CPU, 128Mi~256Mi Mem）、健康检查和节点固定（`lsy-2`）。

**混沌工程。** Chaos Mesh v2.8.1 注入三类故障：StressChaos（CPU 80%/Memory 180MB）和 NetworkChaos（120ms 延迟）。实验使用 fast 模式（25s 混沌/5s 观测）。

### 5.2 基线与消融

| 方法 | 描述 |
|------|------|
| B1-Direct | 直接命令生成 |
| B2-ReAct | 观察-推理-行动循环 |
| B3-Reflexion | ReAct + 反思重试 |
| B4-Template | 模板检索（无安全层） |
| **B5-OpsSkill** | **完整框架** |
| A1-no-IR | 移除类型化 IR |
| A2-no-HiddenVerify | 移除隐藏验证 |
| A3-no-PolicyGate | 移除策略门控 |

### 5.3 主要结果

**表 1：基线对比**

| 方法 | TSR↑ | HVP↑ | UAR↓ | TC↓ | Time | Score↑ |
|------|:----:|:----:|:----:|:---:|:----:|:------:|
| B1-Direct | 100% | 100% | 0.00 | 1.0 | 1.6s | **25%** |
| B2-ReAct | 100% | 100% | 0.00 | 3.0 | 4.3s | **40%** |
| B3-Reflexion | 100% | 100% | 0.00 | 4.0 | 4.4s | **40%** |
| B4-Template | 100% | 100% | 0.00 | 1.1 | 1.4s | **25%** |
| **B5-OpsSkill** | **100%** | **100%** | **0.00** | 2.8 | 7.7s | **90%** |

**核心发现：** OpsSkill（90%）显著领先所有基线（最高 40%），差距源于类型化技能 IR 提供的结构化前置条件检查和验证标准匹配。ReAct 和 Reflexion 表现持平（40%），反思机制未带来提升。模板检索与直接命令等效（25%），说明仅有技能模板缺乏安全层不能利用知识库优势。

| 变体 | Score | ΔScore | UAR↓ |
|------|:-----:|:------:|:----:|

**消融发现：** 类型化 IR 是最关键组件（贡献 65%），隐藏验证贡献 30%，策略门控主要贡献安全性（UAR 0.12→0.00）。

### 5.4 跨域一致性

OpsSkill 在三个故障域（CPU/Memory/Network）上均保持 90% 评分，展示了框架的跨域泛化能力。

### 5.5 安全性分析：变异任务

在唯一涉及集群状态修改的任务 `memory-recover-rollout` 上，B5 通过策略门控阻断变异操作（unsafe=0, score=0.90），而 A3 绕过门控执行了变异（unsafe=1, score=0.85）。策略门控以 1% 评分代价换取零不安全动作率。

### 5.6 执行效率

B5 平均 7.7s（含环境探测 ~3s、前置条件 ~2s、验证 ~2s），高于 B1（1.6s）但低于 10s 阈值。额外时间转化为更高的质量保证。

### 5.7 技能优化前后对比

为验证结构化失败驱动优化器 $u_\omega$ 的有效性，我们设计了 6 个对抗性故障场景，每个场景触发一种特定的失败签名类型 $\sigma$，驱动优化器 $\Delta_\sigma(S)$ 对初始技能 $S_0$ 进行结构化编辑，生成优化后技能 $S_1$。

**实验设计。** 6 个场景覆盖了 6 种核心失败类型：RBAC 权限不足（F1）、目标资源缺失（F2）、Rollout 超时（F3）、就绪探针失败（F4）、CRD 缺失（F5）和工具命令缺失（F6）。

**$\Delta_\sigma$ 失败签名到结构化编辑映射：**

| 失败类型 $\sigma$ | 结构化编辑 $\Delta_\sigma(S)$ | 目标组件 |
|---|---|---|
| RBAC_DENIED | 添加 `kubectl auth can-i` 检查 | 前置条件 |
| RESOURCE_NOT_FOUND | 添加资源存在性探测 | 前置条件 |
| ROLLOUT_TIMEOUT | 添加 `rollout undo` 条目 | 回滚 |
| READINESS_NOT_MET | 添加 pod-readiness jsonpath 信号 | 隐藏检查 |
| CRD_MISSING | 添加 CRD 存在性探测 | 前置条件 |
| COMMAND_NOT_FOUND | 添加工具可用性检查 | 前置条件 |

**优化结果：**

| 场景 | 失败类型 | $S_0$ Score | $S_1$ Score | ΔScore | Precond | Hidden |
|------|----------|:-----------:|:-----------:|:------:|:-------:|:------:|
| F1 | RBAC_DENIED | 0% | 20% | +20% | 1→2 | 2→2 |
| F2 | RESOURCE_NOT_FOUND | 0% | 20% | +20% | 1→2 | 2→2 |
| F3 | ROLLOUT_TIMEOUT | 10% | 45% | +35% | 1→1 | 2→2 |
| F4 | READINESS_NOT_MET | 40% | 70% | +30% | 1→1 | 2→3 |
| F5 | CRD_MISSING | 20% | 20% | +0% | 1→2 | 2→2 |
| F6 | COMMAND_NOT_FOUND | 20% | 20% | +0% | 1→2 | 2→2 |
| **平均** | — | **15%** | **33%** | **+18%** | — | — |

**主要发现：**
1. **平均评分提升 18%**（0.15→0.33）：优化器在 4/6 场景中产生了显著改善。
2. **前置条件增强**：4 个场景的前置条件从 1 增加至 2，新增检查使技能能在执行前检测不兼容环境并安全退出。
3. **隐藏验证信号丰富**：F4 场景新增 pod-readiness 隐藏检查，将假阳性差距从 1.0 降至 0.25。
4. **安全优先**：4/6 场景的改进模式是"将运行时失败转化为编译时阻断"。

## 6 讨论

### 6.1 为什么结构化 IR 比自由文本推理更有效？

B5 与 B2 之间 50 个百分点的差距不是因为执行了更多命令，而是类型化技能 IR 将运维知识编码为可约束、可验证的结构。ReAct 的推理过程以自由文本形式存在，无法被形式化验证（$P_{ratio}=0, V_{vis}=0$）。OpsSkill 将相同知识编译为 $(P,A,V,R)$ 四元组，使每个阶段都有明确判据。

**更广泛的论断：** 在高风险自动化领域，结构化知识表示比端到端生成更可靠。LLM 适合用于*编译*运维知识（$f_\theta$），但执行和验证应由确定性管道完成。

### 6.2 策略门控的安全-效能权衡

A3（89%）仅比 B5（90%）低 1%，但 UAR 从 0% 升至 12%。在生产环境中，一次不安全变异的代价远超 1% 评分——策略门控以极小效能代价换取关键安全保障。

### 6.3 局限性

1. **实验规模**：3 节点集群、8 任务卡、3 故障域；未覆盖大规模集群和更多故障类型
2. **评分函数**：固定权重可能需要按任务类型自适应调整
3. **LLM 依赖**：当前仅使用 heuristic verifier，未启用 LLM verifier
4. **确定性模式**：人工编写的 skill + heuristic verifier 导致评分无方差

### 6.4 对有效性的威胁

- **内部有效性**：评分函数可能偏向结构化输出的方法；通过统一隐藏检查缓解
- **外部有效性**：仅在 K8s 上验证；Typed Skill IR 设计是平台无关的
- **构造有效性**：8 个任务卡由研究者设计；未来需引入真实 incident 数据

---

## 7 结论

本文提出了 OpsSkill，一个面向 Kubernetes 集群的自主运维技能学习框架。通过类型化技能 IR、风险预算门控执行和隐藏多信号验证三层架构，OpsSkill 在 3 个故障域 8 个任务的评估中达到 90% 综合评分，超过所有基线方法 50-65 个百分点，同时保持零不安全动作率。

**主要贡献量化：** 类型化 IR（$-65\%$）> 隐藏验证（$-30\%$）> 策略门控（$-1\%$ 评分，$-12\%$ UAR）。

**未来方向：** LLM 集成深化、大规模验证、跨平台泛化、闭环自优化、人在回路集成。
