# 讨论

## 6 讨论

### 6.1 结果解读与方法论意义

#### 6.1.1 为什么结构化 IR 比自由文本推理更有效？

实验结果揭示了一个关键发现：B5-OpsSkill（90%）与 B2-ReAct（40%）之间 50 个百分点的差距，不是因为 OpsSkill 执行了更多或更复杂的命令，而是因为其**类型化技能 IR** 将运维知识编码为可约束、可验证的结构。具体而言：

- **B2-ReAct** 的"观察-推理-行动"循环虽然能正确推理出需要执行的命令，但其推理过程和执行结果均以自由文本形式存在，无法被形式化验证。其评分被限制在 40%，因为缺乏前置条件（$P_{ratio}=0$）和验证标准（$V_{vis}=0$）的结构化支持。
- **B5-OpsSkill** 通过将相同的运维知识编译为 $(P, A, V, R)$ 四元组，使得每个执行阶段都有明确的成功/失败判据，从而在评分函数的所有维度上获得高分。

这一发现支持了一个更广泛的论断：**在高风险自动化领域（如生产运维），结构化知识表示比端到端生成更可靠**。LLM 的生成能力适合用于*编译*运维知识（$f_\theta$），但执行和验证应当由确定性的受控管道完成。

#### 6.1.2 策略门控的安全-效能权衡

消融实验中 A3（w/o Policy Gate）的结果提供了一个有趣的权衡分析：

- **评分维度**：A3（89%）仅比 B5（90%）低 1%，似乎表明策略门控对任务完成度的影响微乎其微。
- **安全维度**：A3 的 UAR = 0.12（1/8 的任务中执行了不安全变异），而 B5 的 UAR = 0.00。

这一微小的评分差距与显著的安全差距形成了鲜明对比。在生产环境中，一次不安全的变异操作可能导致服务中断、数据丢失或级联故障，其代价远超 1% 的评分损失。因此，策略门控的设计理念是**以极小的效能代价换取关键的安全保障**——这正是论文第 4.5 节形式化风险预算 $B(S,E)$ 的实际价值。

#### 6.1.3 隐藏验证的独特角色

A2（w/o Hidden Verification）的 30% 评分下降揭示了隐藏验证机制的双重角色：

1. **质量保证角色**：隐藏验证（$V_{hid}$）提供了与技能内部验证（$V_{vis}$）独立的第二层质量检查，防止了"自验证过拟合"——即技能将验证标准设计为总是通过的退化情况。
2. **评估公平性角色**：在 baseline 对比中，所有方法都接受相同的隐藏检查，确保评分不受各方法内部验证策略差异的影响。

### 6.2 与现有方法的关系

#### 与 ReAct / Reflexion 的比较

ReAct \cite{yao2022react} 和 Reflexion \cite{shinn2023reflexion} 代表了当前 LLM-based agent 的主流范式。在通用推理和工具使用任务上，它们展示了强大的适应性。然而，我们的实验表明，在运维任务中，这些方法的"自由推理 + 自由行动"模式面临两个根本限制：

1. **缺乏执行前安全审查**：ReAct 在推理后直接执行动作，没有策略门控机制来阻断高风险操作。
2. **缺乏结构化验证**：Reflexion 的"反思"基于自由文本的执行结果摘要，而非形式化的验证标准匹配。

OpsSkill 保留了 LLM 的生成和推理能力（用于技能编译 $f_\theta$ 和 LLM verifier $h_\phi$），但将执行和验证放入受控管道——这种"LLM 生成 + 确定性执行"的混合架构可能是高风险自动化领域的通用设计模式。

#### 与模板/Runbook 系统的比较

B4-Template 与 B1-Direct 的等分（25%）说明了传统 Runbook 系统的局限性：**即使拥有正确的运维操作模板，如果没有安全层（前置条件 → 策略门控 → 验证），模板的质量优势无法转化为执行质量**。OpsSkill 的类型化 IR 可以看作是"带安全约束的智能 Runbook"，将传统 Runbook 的确定性优势与 LLM 的适应性相结合。

### 6.3 局限性

#### 6.3.1 实验规模

当前实验在单一 Kubernetes 集群（3 节点）上使用 8 个任务卡和 3 个故障域进行评估。虽然任务覆盖了检测、诊断、恢复和验证四个运维阶段，但仍有以下局限：

- **集群规模**：3 节点集群无法代表大规模生产环境（100+ 节点）的复杂性。在大规模集群中，故障传播模式、资源竞争和命名空间隔离的挑战可能影响 skill 的可移植性。
- **故障类型**：仅评估了 CPU 压力、内存压力和网络延迟三种故障类型。真实生产环境中还可能遇到磁盘 I/O、DNS 故障、证书过期、配置漂移等更复杂的故障模式。
- **单应用目标**：实验中仅使用了一个 nginx Deployment 作为目标。多服务架构中的故障诊断（需要跨服务追踪）可能需要更复杂的 skill 组合。

#### 6.3.2 评分函数设计

当前综合评分函数使用固定权重（$w_P=0.20, w_A=0.20, w_{V_{vis}}=0.25, w_{V_{hid}}=0.25, w_R=0.10$），这些权重基于运维实践经验设定，但在不同运维场景中可能需要调整。例如，在纯监控任务中，前置条件检查的权重可能应当更高；在恢复任务中，回滚机制的权重可能更重要。未来工作可以探索基于任务类型的自适应评分权重。

#### 6.3.3 LLM 依赖性

当前实验使用 heuristic verifier（基于退出码和 stdout 匹配），未启用 LLM verifier。在更复杂的验证场景中（如判断日志输出是否表明根因已被定位），LLM verifier 可能提供更准确的语义判断。但同时，LLM verifier 引入了额外的延迟和不确定性，需要在准确性和效率之间进行权衡。

#### 6.3.4 确定性评分模式

当前实验中各方法的评分在所有故障域上完全一致（如 B5 在三个域上均为 90%），这是因为所有 skill 都是人工精心编写的，且使用确定性 heuristic verifier。在 LLM 参与编译或验证的场景中，预期会出现评分方差，这需要多次重复实验和统计显著性检验。

### 6.4 对有效性的威胁

#### 内部有效性

- **评分函数的公平性**：评分函数的设计可能偏向于具有更多结构化输出的方法（如 B5）。我们通过确保所有方法接受相同的隐藏检查（HVP）和使用统一的评分维度来缓解这一威胁。
- **实现偏差**：各 baseline 的实现可能存在优化差异。我们确保每个 baseline 都使用其方法论的标准实现（如 ReAct 使用标准的 observe-reason-act 循环）。

#### 外部有效性

- **集群类型**：实验在 Kubernetes 集群上进行，结论可能不直接适用于非 Kubernetes 环境（如虚拟机集群、Serverless 平台）。但 Typed Skill IR 的设计是平台无关的，核心概念（前置条件、动作、验证、回滚）适用于任何自动化执行环境。
- **故障真实性**：使用 Chaos Mesh 注入的人工故障可能不完全代表生产环境中的真实故障模式。但 Chaos Mesh 已被广泛用于工业界的混沌工程实践，其故障模型具有一定的代表性。

#### 构造有效性

- **任务卡设计**：8 个任务卡均由研究者设计，可能无法覆盖所有运维场景。未来工作应引入来自工业界的真实 incident 数据来构造更具代表性的 task benchmark。
