# 相关工作初稿（中文）

现有与 OpsSkill 最相关的研究主要分为四类：AIOps 与事故知识挖掘、微服务诊断与根因定位、运行手册/故障排查指南自动化，以及通用 Agent 工具调用评测与安全治理。

第一类工作关注如何从真实云服务事故与运维调查中提取结构化知识。例如，DeCaf 将大规模云服务中的性能问题诊断与分诊建模为一个连续流程，说明真实运维并非单一预测任务，而是由多个相互依赖的处置步骤组成。后续工作进一步尝试从云服务 incident 调查记录中挖掘根因知识，或从事故文本中抽取结构化运维知识。这类研究为 OpsSkill 提供了重要启发：历史运维记录和故障调查过程可以转化为机器可利用的知识来源。然而，这些方法大多停留在知识抽取、故障模式识别或分诊层面，尚未将这些知识编译为带有前置条件、回滚与成功判据的可执行技能。

第二类工作聚焦微服务系统中的诊断与根因定位，如细粒度性能诊断、可解释故障定位以及基于异构图或不完整指标的根因分析方法。这些研究显著提升了定位故障原因的能力，也强调了“actionable”和“interpretable”的重要性。但总体而言，这些方法主要解决的是“找出问题在哪里”，而不是“如何在真实受限环境中安全地执行修复动作并验证修复是否真正有效”。换言之，它们更像是自治运维系统中的诊断模块，而非完整的技能学习与执行框架。

第三类工作是与 OpsSkill 最接近的方向，即 troubleshooting guide、runbook 和 incident copilot 的自动化。例如，AutoTSG 研究了如何学习并合成 incident troubleshooting 过程；近期还有工作将人类编写的 troubleshooting guide 或 runbook 融入 agentic support automation 或 incident mitigation copilot 中。这些研究说明，将运维经验和排障流程转化为可操作计划是可行且有价值的。但现有方法通常缺乏统一的结构化 skill 表示，也很少在真实远程 Kubernetes 环境中系统考察隐藏验证、安全门控、回滚质量以及跨环境迁移问题。

第四类工作涉及通用 Agent 工具调用评测与自治安全治理。一方面，现有 benchmark 开始关注大模型在工具调用、多步交互和状态化环境中的能力；另一方面，AIOps 研究也开始讨论自动化等级、治理机制以及人在回路的重要性。这些工作为 OpsSkill 提供了评测和安全设计上的背景，但它们要么聚焦通用工具调用，而非真实云原生运维任务；要么停留在原则性讨论，缺少针对远程 Kubernetes 运维技能的统一评测框架。

因此，与现有工作相比，OpsSkill 的目标不是构建一个通用问答式运维助手，而是提出一种面向真实远程 Kubernetes 环境的自治运维技能学习框架：它能够将任务自动编译为结构化技能，在策略约束下安全执行，并通过隐藏的多信号验证器判断任务是否真正完成，进一步利用结构化执行报告持续优化技能质量。
