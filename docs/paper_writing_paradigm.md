# 可复用的科研论文完成范式（从 OpsSkill 项目中提炼）

## 0. 目标
将一个研究从 **文献调研 → 创新点提炼 → 方法设计 → 代码实现 → 实验验证 → 论文撰写 → 审稿模拟 → 反复修改 → 提交** 全流程打通，所有产物可复现。

---

## 全流程概览

```
阶段 1  文献调研与领域洞察
  ↓
阶段 2  创新点提炼与研究定位
  ↓
阶段 3  问题定框（研究契约）
  ↓
阶段 4  方法形式化
  ↓
阶段 5  代码实现（方法落地）
  ↓
阶段 6  实验设计
  ↓
阶段 7  实验执行与数据流水线
  ↓
阶段 8  论文撰写
  ↓
阶段 9  证据-声称对齐
  ↓
阶段 10  自审稿与模拟评审
  ↓  ← 修改循环（回到阶段 4–9 中的薄弱环节）
阶段 11  排版与多版本转换
  ↓
阶段 12  鲁棒性加强包
  ↓
阶段 13  提交就绪包
```

---

## 阶段 1：文献调研与领域洞察

> 这是一切的起点。不了解别人做了什么，就不可能知道自己能做什么。

**目标：** 系统扫描目标领域，建立知识地图，找到研究缺口。

**操作步骤**
1. **确定搜索关键词**：围绕目标问题，列出 3–5 组核心关键词和同义词
2. **多源搜索**：
   - Google Scholar / Semantic Scholar（广度）
   - DBLP / ACL Anthology / IEEE Xplore / ACM DL（按期刊/会议定向搜）
   - arXiv（最新预印本，看趋势）
   - Connected Papers / Litmaps（引用图谱，发现关键节点）
3. **筛选与分层**：
   - 高相关（必读）：10–20 篇
   - 中相关（了解）：20–30 篇
   - 低相关（备查）：列表即可
4. **精读高相关论文**，提取：
   - 每篇的核心方法（一句话）
   - 每篇的评估方式（用了什么数据集/指标）
   - 每篇的明确局限（作者自述 + 你的判断）
   - 每篇的代码/数据是否可复现

**产出物**
- 文献调研表（见下方模板）
- 领域知识地图（哪些子方向已饱和、哪些有空白）
- 关键论文引用图谱

**文献调研表模板**

| 编号 | 论文 | 年份 | 会议/期刊 | 核心方法 | 评估方式 | 优点 | 局限 | 与我的关系 |
|------|------|------|----------|---------|---------|------|------|----------|
| P01  | ...  | 2024 | ICSE     | ...     | ...     | ...  | ...  | 直接竞品  |
| P02  | ...  | 2023 | NeurIPS  | ...     | ...     | ...  | ...  | 方法借鉴  |

**检查清单**
- [ ] 至少读了 10 篇高相关论文的全文
- [ ] 能口头说出该领域「做到哪了」「还缺什么」
- [ ] 文献覆盖了最近 3 年的核心会议/期刊
- [ ] 已识别 2–3 个研究缺口（gap）

## 阶段 2：创新点提炼与研究定位

> 创新不是凭空发明，而是在已有工作的缝隙中找到有价值的新角度。

**目标：** 从文献缺口中提炼出明确、可执行的创新点，确定论文要写什么。

**创新点发现的五种模式**

| 模式 | 描述 | 示例（OpsSkill） |
|------|------|------------------|
| **缺失环节** | 现有流水线中某一步没人做 | 运维知识有人提取，但没人编译成可安全执行的技能 |
| **弱假设** | 现有方法依赖不现实的假设 | ReAct 假设命令成功=任务完成，实际不是 |
| **组合创新** | 把两个领域的成熟技术组合 | 类型系统（PL）+ 运维自动化（DevOps） |
| **度量缺陷** | 现有评估不公平或不全面 | 各方法用不同公式打分，无法横比 |
| **场景迁移** | 把 A 场景的方法搬到 B 场景 | Reflexion 用于对话，我们用于运维 |

**操作步骤**
1. 从文献调研表的「局限」列中，归纳出 2–3 个高频缺口
2. 对每个缺口，问：
   - 这个缺口是否影响实际使用？（实用性）
   - 现有技术是否有潜力解决？（可行性）
   - 解决后是否能量化改进？（可度量性）
3. 选择同时满足「实用 + 可行 + 可度量」的缺口作为主攻方向
4. 将创新点写成「A 做不到 X，因为缺少 Y，我们通过 Z 来解决」的句式
5. 反向验证：这个创新点能不能拆成 3–5 条具体贡献？

**产出物**
- 创新点陈述（一段话，含动机、方法、预期效果）
- 贡献列表（3–5 条，每条可度量）
- 方法草案（非正式，但说清核心思路）
- 目标期刊/会议及其偏好（理论型 vs. 系统型 vs. 应用型）

**检查清单**
- [ ] 创新点不是「换了个数据集跑已有方法」
- [ ] 能用一句话说清「我们做了什么新东西」
- [ ] 每条贡献有对应的验证手段（实验 / 证明 / 案例）
- [ ] 目标期刊/会议的 scope 覆盖这个主题

---

## 阶段 3：问题定框（研究契约）

**产出物**
- 一句话问题陈述
- 范围边界（做什么 / 不做什么）
- 3–5 条可度量的贡献
- 研究问题（RQ），每个 RQ 直接对应一个实验

**检查清单**
- [ ] 问题是否来自真实痛点？
- [ ] 贡献是否可测试（不只是概念性的）？
- [ ] RQ 能否直接映射到实验设计？

---

## 阶段 4：方法形式化（从想法到可验证的方法）

**产出物**
- 系统架构图
- 形式化定义（状态、输入、输出、约束）
- 核心目标函数
- 至少一个安全性/正确性命题

**检查清单**
- [ ] 每个模块有明确的输入输出
- [ ] 公式中的变量与代码实现一一对应
- [ ] 声称的性质有证明草案或实验代理指标

---

## 阶段 5：代码实现（方法落地）

> 形式化只是纸上谈兵，代码才是真正的检验。

**目标：** 将方法设计转化为可运行、可测试、可复现的代码。

**操作步骤**
1. **先搭骨架**：根据架构图，建立模块目录和接口定义
2. **核心模块优先**：先实现贡献最大的组件（参照消融预期）
3. **单元测试随写随测**：每个模块完成后立即验证
4. **冒烟测试**：端到端跑通一个最简单的场景
5. **配置外置**：所有环境参数、超参数写入配置文件

**产出物**
- 可运行的代码包（含清晰的模块划分）
- 配置文件（环境 + 超参数）
- 冒烟测试脚本（一键验证基本功能）
- 依赖清单（requirements.txt / environment.yml）

**检查清单**
- [ ] 代码结构与架构图一一对应
- [ ] 核心模块有单元测试
- [ ] 冒烟测试通过
- [ ] 换一台干净机器能跑起来
- [ ] 代码中的变量名与论文公式中的符号有对应关系

---

## 阶段 6：实验设计（先规划再执行）

**产出物**
- 基线矩阵（每个基线为什么被选中）
- 消融计划（组件级因果推理）
- 场景/任务分类法（领域覆盖）
- 统一度量定义 + 敏感性分析计划

**检查清单**
- [ ] 主结果、消融、鲁棒性各有独立小节
- [ ] 指标同时包含有效性和安全性/开销
- [ ] 至少有一组对抗/边界场景实验

---

## 阶段 7：实验执行与数据流水线（可复现优先）

**产出物**
- 环境配置文件
- 一键实验运行脚本
- 结构化结果 JSON
- 自动生成表格/图表的脚本

**检查清单**
- [ ] 一条命令可重新生成关键结果
- [ ] 保留原始数据（不仅是最终表格）
- [ ] 失败日志有结构化签名

---

## 阶段 8：论文撰写（分层起草）

**推荐写作顺序**
1. 方法（Method）
2. 实验（Experiments）
3. 引言（Introduction）
4. 相关工作（Related Work）
5. 讨论与局限（Discussion）
6. 结论（Conclusion）
7. 摘要（Abstract）——最后写

**产出物**
- 按小节的草稿文件
- 合并后的完整草稿
- 图表标题自包含（不看正文也能看懂）

**检查清单**
- [ ] 每条贡献在引言、方法、实验、结论中都出现
- [ ] 每张图回答一个具体的 RQ
- [ ] 每张表在正文中有一句总结性解读

---

## 阶段 9：证据-声称对齐（关键卡口）

建立映射表：

| 声称 | 证据（图/表/证明） | 强度 | 是否有缺口？ |
|------|-------------------|------|------------|
| C1   | 表1 + 图2         | 强   | 无         |
| C2   | 命题1             | 强   | 无         |
| ...  | ...               | ...  | ...        |

**硬性规则：没有直接证据的声称不得出现。**

---

## 阶段 10：自审稿与模拟评审

> 不要等真正的审稿人来发现问题。自己先当最严格的审稿人。

**目标：** 站在评审人角度审视论文，提前修补薄弱环节。

**10a. 自我审查清单**

| 审查维度 | 常见问题 | 检查方法 |
|---------|---------|--------|
| 新颖性 | 「这和 XX 有什么区别？」 | 写出与每个最近似工作的 3 点差异 |
| 技术深度 | 「证明不严谨」「缺形式化」 | 检查每个命题的假设是否明确、证明是否完整 |
| 实验充分性 | 「基线太少」「没有消融」 | 检查是否有 ≥4 个基线 + 完整消融 |
| 可复现性 | 「代码没开源」「参数没给」 | 检查是否有可运行的代码 + 完整配置 |
| 写作清晰度 | 「看不懂」「符号不统一」 | 让非本方向的同学读引言，看能否说出你做了什么 |
| 公平性 | 「指标偏向自己」 | 检查是否有统一评估 + 敏感性分析 |

**10b. AI 模拟评审**

将论文（或摘要+方法+实验摘要）发给 AI，要求它扮演目标期刊/会议的严格审稿人，给出：
- 3 个优点
- 3 个缺点（要求严格但公正）
- 2 个会问作者的问题
- 1 个致命缺陷（如有）
- 1 个能加强论文的额外实验建议

**10c. 修改循环**

```
发现薄弱点 → 定位到具体阶段（阶段 4–9）
  → 补实验 / 补证明 / 补形式化 / 重写段落
  → 重新生成图表
  → 重新检查证据-声称映射
  → 再次模拟评审
  → 循环直到无「无支撑的强声称」
```

**常见修改模式**

| 审稿意见类型 | 回到哪个阶段 | 典型动作 |
|-------------|-------------|--------|
| 「和 XX 方法有什么区别？」 | 阶段 2（创新点） | 补充差异分析表 |
| 「实验不够充分」 | 阶段 6–7（实验） | 加基线 / 加数据集 / 加消融 |
| 「形式化不严谨」 | 阶段 4（方法） | 补假设、补证明、补符号定义 |
| 「写作不清晰」 | 阶段 8（写作） | 重组段落结构、加过渡句 |
| 「缺乏实用性论证」 | 阶段 6（实验） | 加案例分析 / 加效率实验 |
| 「评估不公平」 | 阶段 12（鲁棒性） | 加统一指标 + 敏感性分析 |

**检查清单**
- [ ] 至少做了 2 轮自审稿
- [ ] 模拟审稿的每个缺点都有对应修改
- [ ] 修改后重新通过证据-声称对齐检查
- [ ] 能写出每个潜在质疑的回复（rebuttal 草案）

---

## 阶段 11：排版与多语言/多版本转换

**产出物**
- 符合目标期刊/会议格式的 LaTeX 版本（如 IEEE 双栏）
- 可选的中英双语版 / 完整版
- 版本间一致性检查报告

**检查清单**
- [ ] 翻译/压缩过程中无静默内容丢失
- [ ] 所有图表被引用且存在
- [ ] 编译干净（或仅有可接受的轻微警告）

---

## 阶段 12：鲁棒性与公平性加强包（强烈建议）

至少包含：
- 统一评估指标（对所有方法用同一公式）
- 权重敏感性分析
- 逐任务/逐域细粒度分解
- 包含失败和恢复路径的案例分析

这些能预防「指标挑选偏好」的审稿批评。

---

## 阶段 13：提交就绪包

**最小提交包**
- 论文源码 + PDF
- 可复现脚本
- 结果表格/图表
- README（含重跑说明）
- 公开代码仓库（如允许）

## 实用模板（可直接迁移到其他论文）

### A. 项目目录模板
```
项目名/
├── docs/           # 论文、草稿、参考文献
├── experiments/    # 任务卡、场景配置、故障注入
├── scripts/        # 运行实验 + 生成表格/图表
├── results/        # 原始 JSON + 最终 tables/ figures/
├── src/ 或 包目录/  # 方法实现
├── configs/        # 环境配置
├── skills/ 或 models/ # 学习到的产物
└── README.md
```

### B. 论文章节骨架
1. 引言（Introduction）
2. 相关工作（Related Work）
3. 问题定义（Problem Definition）
4. 方法（Method）
5. 实验（Experiments）
6. 讨论（Discussion：局限性 + 效度威胁）
7. 结论（Conclusion）

### C. 核心实验套件
- 主对比实验（Main Comparison）
- 消融实验（Ablation）
- 跨域/跨场景分析（Cross-domain）
- 效率/安全分析（Efficiency / Safety）
- 鲁棒性/敏感性分析（Robustness / Sensitivity）
- 案例分析（Case Studies）

---

## 从 OpsSkill 提炼的元原则

1. **结构胜于自由文本**：把方法和评估编码为有类型的产出物。
2. **安全是一等指标**：不只看最终得分，还要看安全率。
3. **失败是资产**：把失败签名转化为方法改进。
4. **公平需要统一指标**：避免方法专属的评分偏差。
5. **鲁棒性是必须的**：敏感性和逐任务分析加固结论。
6. **版本一致性要守住**：翻译/排版版本必须保留实质内容。

## 五阶段执行计划（适用于新论文）

| 执行阶段 | 时间 | 对应范式阶段 | 内容 |
|---------|------|-------------|------|
| 调研期 | 第 1–5 天 | 阶段 1–2 | 文献调研、创新点提炼、研究定位 |
| 设计期 | 第 6–10 天 | 阶段 3–4 | 问题定义、方法形式化、实验协议 |
| 开发期 | 第 11–20 天 | 阶段 5–7 | 代码实现、跑实验、自动生成图表 |
| 写作期 | 第 21–28 天 | 阶段 8–9 | 写全文、对齐声称与证据 |
| 打磨期 | 第 29–35 天 | 阶段 10–13 | 模拟审稿、修改循环、排版、提交 |

**原则：**
- 调研期不扎实 → 创新点经不起推敲
- 开发期不扎实 → 实验数据不可信
- 跳过打磨期 → 审稿人替你发现问题

---

## 完成定义（Definition of Done）

一篇论文「完成」意味着：
- [ ] 所有主要声称有明确的证据支撑
- [ ] 消融实验和鲁棒性分析均已完成
- [ ] 复现路径已文档化且可运行
- [ ] 排版/翻译版本与源版本内容一致
- [ ] 外部审稿人无需隐含假设即可跟上逻辑
