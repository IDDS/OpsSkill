# 引言初稿（中文）

## 1 引言

随着云原生系统和微服务架构在工业界的广泛部署，DevOps 与 SRE 团队需要持续处理大量重复但高风险的运维任务，例如故障诊断、配置修复、服务恢复、扩缩容调整以及权限或依赖异常处理。这些任务通常依赖运维人员对系统状态、工具链、故障模式以及组织约束的综合理解，具有明显的状态依赖性、环境异构性和执行风险。近年来，大语言模型在推理、代码生成与工具调用方面的快速进展，使得构建能够辅助甚至部分自动完成运维任务的自治 Agent 成为一个极具吸引力的研究方向。

然而，现有方法距离真实可用的自治运维仍存在显著差距。一方面，许多方法将运维任务简化为自由形式的命令生成或交互式工具调用问题，忽视了运维动作本身具有强约束和高代价的特点。对于真实生产或准生产环境而言，错误的命令序列不仅会导致任务失败，还可能带来跨命名空间影响、服务中断甚至不可逆的数据损坏。另一方面，现有评测往往依赖可见的命令返回值或单一检查项来判断任务是否完成，但在真实系统中，“命令执行成功”并不等价于“服务真正恢复”。例如，Pod 被成功重启并不意味着依赖链路恢复，Deployment rollout 完成也不意味着用户侧错误率已经下降。因此，自治运维系统不仅需要会“调用工具”，更需要具备安全执行、隐藏验证与失败后优化的能力。

AIOps 与云服务事故分析领域的已有工作为这一问题提供了重要基础。已有研究表明，云服务 incident 记录、故障调查过程和 troubleshooting guide 中蕴含着丰富的可复用知识；另一些工作则在微服务诊断、根因定位和 runbook 自动化方面取得了进展。这些研究证明了将运维经验结构化、自动化的可行性。然而，大多数现有方法要么停留在诊断、知识抽取和分诊层面，要么缺少对技能执行语义的统一建模，尚未形成一个覆盖“技能生成—安全执行—真实验证—持续优化”的闭环框架。尤其是在远程 Kubernetes 这类真实且受限的环境中，如何将任务自动编译为可执行、可回滚、可评测的技能，并在保证风险可控的前提下进行验证，仍是一个开放问题。

为解决上述问题，本文提出 OpsSkill，一个面向远程 Kubernetes 环境的自治运维技能学习框架。OpsSkill 的核心思想是：将运维任务从一次性的命令生成问题提升为结构化技能学习问题。具体而言，我们将一个运维任务编译为带类型的技能表示，包含前置条件、动作序列、成功判据、回滚计划和风险标签；在执行层面，设计策略约束与能力探测机制，对技能进行门控执行；在评测层面，引入多信号隐藏验证器，综合对象状态、事件、日志与服务健康信息判断任务是否真正完成；在优化层面，利用结构化执行报告和失败签名持续改进技能质量，而不仅依赖自由文本反思。通过这样的设计，OpsSkill 不再将自治运维视为“给定问题后直接输出命令”的黑盒生成任务，而是视为“在约束环境中学习、验证和改进可复用技能”的系统问题。

本文的研究目标主要体现在以下三个方面。第一，我们希望提出一种适用于运维场景的统一技能抽象，使任务表示、执行过程和评测结果可以在同一框架中被描述和分析。第二，我们希望构建一套面向远程 Kubernetes 的真实验证机制，使评估不再局限于表面可见输出，而能够考察任务对系统真实状态的影响。第三，我们希望通过策略门控和结构化失败优化机制，在保持较高成功率的同时降低危险操作比例，推动自治运维从“能跑通 demo”走向“能在真实环境中受控运行”。

基于上述目标，本文做出如下贡献：

1. 提出一种面向运维任务的 Typed Ops Skill IR，将任务卡编译为包含前置条件、动作序列、成功判据、回滚计划与风险标签的结构化技能表示，从而为自治运维提供可审计、可复用、可验证的统一抽象。
2. 设计一种面向远程 Kubernetes 的策略门控执行框架，在技能执行前进行能力探测、范围约束与风险检查，在执行中支持回滚触发，从而降低不安全动作与不可恢复失败的概率。
3. 提出一种面向真实远程环境的隐藏多信号验证机制，将可见检查与隐藏检查相结合，更准确地判断技能是否真正完成运维目标。
4. 提出一种基于结构化执行报告与失败签名的技能优化机制，使系统能够从失败中持续学习并提升技能稳定性与可迁移性。
5. 计划在真实远程 Kubernetes 集群上构建面向诊断、恢复、配置修复与权限受限任务的实验集合，系统评估所提方法在隐藏成功率、安全性、回滚质量和跨环境迁移等方面的表现。

本文余下部分安排如下：第 2 节介绍相关工作；第 3 节给出问题定义与技能表示；第 4 节详细描述 OpsSkill 的方法设计；第 5 节介绍实验设置；第 6 节报告实验结果；第 7 节讨论方法局限与安全问题；最后总结全文并展望未来工作。
